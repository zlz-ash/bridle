# 工具系统整改计划

## 已确认原则

- 模型可以把任意命令交给候选容器执行；安全控制属于容器边界，不能依赖工具层命令白名单。
- 任意探索命令只产生诊断结果，不能改变正式测试门禁状态；只有冻结测试合同中的验收命令及其稳定命令 ID 可以产生有效红、最终绿和发布资格。
- 测试由工作流状态自动触发：测试合同通过并标记“允许红”后自动运行红测，候选代码提交后自动运行最终测试。
- “开发前的绿”指测试合同审核通过，不指测试执行通过。对既有项目行为进行修改时采用先红再绿：新增目标测试应按预期失败，原有基线测试仍应保持绿色。
- 先红再绿只作为项目修改流程中的验证要求，不扩张为所有命令执行、项目初始化或无既有行为基线场景的统一前提。
- 模型通过候选容器 Bash 直接修改代码；不新增补丁应用 CLI，也不保留必须逐次提交补丁的修改协议。候选提交时比较候选目录与冻结基线，按需生成最终 diff 并据此发布。
- 统一代码观测做成候选容器 CLI；项目计划、Mail、候选和验证诊断做成控制面 CLI。CLI、模型工具和 API 必须调用同一套应用服务，不能复制业务逻辑。
- 全面观测指状态、边界、结果和失败原因完整可追踪，不等于保存所有原始内容。持久化只保留门禁所需标识、哈希、变更路径、结果摘要、错误码和必要产物引用；不重复保存逐次编辑补丁、完整源码或大段最终 diff，完整差异由基线、候选或 Git 按需生成。
- `execute_plan_node` 是计划节点唯一执行入口，并替代只确认既定节点的 `select_node`。工具调用只创建或复用后台节点工作流并立即返回持久化 wait 信号，不等待测试、编码、Review 或发布结束。
- wait 信号只有 `waiting` 与 `ended` 两种状态，工作流阶段和终态结果分别记录；测试失败、Review 拒绝等内部循环期间信号保持 `waiting`，只有整条工作流成功、失败、阻塞或取消时才变为 `ended`。
- 主 Agent 只有在节点工作流进入终态后才通过多智能体 Mail 收到结果；持有 wait ID 时可以主动查询，也可以完全不查询。查询不推进、不重试、不取消工作流，未查询也不影响后台执行和最终消息投递。
- 所有整改必须先盘点并改造现有实现：老的能用要先用，能改要先改；现有 `AgentToolRegistry`、`ProjectPlanStore`、`PersistentMailbox`、`CandidateService`、`PlanService`、`bridle/cli.py` 和 outbox 等组件能承载新语义时必须原位演进，只有存在明确且可验证的能力缺口时才允许新增组件。
- 禁止同一能力在项目中长期并存多个版本：不得通过 `v2`、`new_*`、旁路 store/service/schema、双写双读或长期兼容分支绕过旧实现。新接口接管调用方的同一实施批次必须删除旧 schema、handler、状态、配置、测试和孤儿代码；持久化数据迁移可以短暂兼容，但必须定义退出条件，并在迁移验证完成后删除兼容路径。

## 每项实施约束

1. 开始实现前，先定位该能力现有的事实源、服务、schema、handler、生产调用方和测试，形成“复用、原位改造、删除”清单；没有完成旧实现盘点，不得新增同类组件。
2. 优先在现有实现上原位改造，并先把生产调用方迁移到统一入口；不得先造一套平行实现，再把旧版本无限期保留为兼容路径。
3. 新语义通过该计划项测试后，在同一实施批次删除被替代的旧入口及其孤儿代码、配置和测试。确需迁移持久化数据时，只允许保留有明确迁移条件、完成标志和删除时点的一次性兼容逻辑。
4. 每项完成验收必须证明生产调用图中该能力只剩一个入口、一个事实源和一套状态语义；使用 CodeGraph callers/impact 核对结构调用关系，使用文本搜索检查遗留名称，并执行该计划项列出的测试。只要仍存在并行版本、双写双读或无人调用的旧路径，该计划项就不能标记完成。

## 未完成计划

- [ ] 5. 补全候选提交、打回、再提交与原子发布生命周期
  - 原因：候选结果目前偏向一次性 `ready/blocked`，不足以表达开发完成后提交、测试失败打回、继续修改、再次提交和最终发布的循环。
  - 修改：模型通过候选容器 Bash 直接修改候选目录，不新增补丁应用 CLI。提交时冻结候选代码哈希，并比较候选目录与冻结基线得到变更路径和最终 diff；最终验证只验证该冻结版本。失败时保留候选工作区和必要诊断并退回 `IMPLEMENTING`；通过时进入 `READY_TO_PUBLISH`，随后重新校验基线并用当次计算结果原子发布。
  - 保留：最终测试失败不直接污染主工作区，不销毁仍需修复的候选；现有文件写集、只读边界和原子提交约束继续生效。中间编辑过程不要求转换成统一补丁，也不记录逐次补丁内容。
  - 完成标准：验证中的候选不可被静默修改；失败结果能准确关联候选代码哈希；再次提交冻结新版本并重新验证；发布所用差异必须由冻结基线与冻结候选计算，不能接受模型自报 diff；只有最终绿且基线未过期的候选可以发布；冲突时拒绝发布并要求重新基线化。
  - 测试：覆盖 Bash 多次直接编辑后的最终差异计算、新增/修改/删除/重命名文件、提交冻结、验证期间修改拒绝、失败打回、再次提交成功、基线漂移、发布冲突和原子发布失败回滚；断言无需补丁 CLI 或逐次补丁记录也能正确发布。
  - 验证：`python -m pytest backend/tests/unit/agent/container/test_candidate_submission.py -q`

- [ ] 6. 强制所有模型命令进入候选容器，并隔离探索结果与权威结果
  - 原因：允许任意命令的安全前提是容器成为唯一执行边界；当前 `backend/src/bridle/agent/tools/executor.py` 仍存在没有容器后端时在宿主机执行的路径。
  - 修改：正式 agent 执行路径缺少候选容器时直接返回明确错误，不回退宿主机；容器统一约束挂载、工作目录、非 root 用户、网络、CPU、内存、进程数、超时、输出、环境变量、密钥和清理。为每次命令运行标记 `exploratory` 或 `authoritative`，后者只能由自动验证协调器依据冻结合同创建。
  - 保留：命令内容本身不做工具层白名单裁剪；容器内仍允许模型执行构建、调试、局部测试和诊断命令。
  - 完成标准：任何模型命令都不能落到宿主机；探索命令即使成功也不能将项目标绿；权威命令必须绑定测试合同、候选版本和稳定命令 ID；容器销毁后不存在越界写入或遗留进程。
  - 测试：覆盖无容器后端拒绝、任意 shell 命令在容器内成功、宿主机路径不可见、网络开关、资源和超时限制、输出截断、密钥脱敏、探索命令不能改变状态，以及权威命令来源校验。
  - 验证：`python -m pytest backend/tests/unit/agent/container/test_container_command_boundary.py -q`

- [ ] 7. 建立连续、持久化且可校验的验证证据链
  - 原因：红测、代码修改和最终绿测如果不能绑定到同一测试合同、候选版本与容器环境，就无法证明最终代码满足最初验收要求。
  - 修改：扩展 `backend/src/bridle/agent/container/candidate_contract.py` 的持久化结果，只记录门禁需要的 `run_id`、`node_id`、`candidate_id`、合同哈希、测试与候选代码哈希、必测命令 ID、`map_seq`、边界与镜像身份、退出码、时间、分类结果、变更路径摘要以及必要诊断产物引用；状态迁移引用不可变证据 ID。
  - 保留：日志和证据中的密钥、令牌与敏感环境变量必须脱敏；失败诊断和权威测试输出可以通过受控产物引用按需保存，但不持久化探索命令的完整输出，不重复保存逐次编辑补丁、完整源码或大段最终 diff。
  - 完成标准：可以从发布结果反向追溯最终绿、有效红、测试合同和候选代码哈希；需要查看完整代码差异时从基线与候选或 Git 生成；缺少任何必测命令、证据版本不一致或必要产物校验失败都不能发布；重启后最小证据仍可查询和验证。
  - 测试：覆盖最小证据完整链、缺失必测命令、合同与代码哈希不匹配、必要产物篡改、脱敏、跨重启读取，以及大量代码修改时不把完整 diff 或源码写入日志与证据表；断言发布门禁使用持久化证据而不是内存标志。

- [ ] 8. 完善失败分类、自动重试、并发失效与全面观测
  - 原因：业务测试失败、测试合同错误、容器故障、基线过期和安全越界需要不同处理；统一标记失败会导致错误打回、无限重试或证据丢失。
  - 修改：为验证协调器定义失败码和重试策略：业务失败打回实现，合同无效退回测试编写，临时基础设施故障有限自动重试，基线或边界变化使证据失效，安全违规立即阻断并审计。为合同审核、允许红、命令调度、容器身份、红色分类、提交、最终门禁、打回、发布与清理记录结构化事件和指标。
  - 保留：重试只适用于明确的临时基础设施故障，不能自动重试业务失败直到偶然通过；观测事件不替代权威证据，也不能改变状态机结果。
  - 完成标准：每类失败都有唯一处理路径、有限重试预算和最终状态；并发修改会使旧证据失效；能够按一次修改任务完整查看状态时间线、命令结果、资源使用、打回原因、发布结果和清理结果。
  - 测试：覆盖业务失败不重试、临时容器故障重试后成功、重试耗尽、基线并发变化、边界或镜像升级、安全违规阻断、日志脱敏和完整状态时间线。

- [ ] 9. 建立共享应用服务之上的 CLI 与模型工具薄适配层
  - 原因：当前模型工具 schema、运行时 handler 和内部服务存在重复或半注册状态；把确定性能力做成 CLI 可以由 Bash、人工和 CI 复用，但如果 CLI 重写业务逻辑又会形成新的事实源分裂。
  - 修改：扩展 `backend/src/bridle/cli.py` 的 Typer 命令树。候选容器只提供 `bridle code inspect/search/graph` 代码观测 CLI；控制面提供 `bridle plan`、`bridle agent`、`bridle candidate` 和 `bridle verify` 命令组，并分别复用项目地图、Mail、候选与验证应用服务。模型的通用命令能力直接连接候选容器 Bash，不再包装成 `bridle run-command`；不实现 `bridle patch`，也不把补丁应用做成 CLI。
  - 保留：候选容器不得直接挂载或任意写入控制面 SQLite 数据库；控制面 CLI 的 agent 身份和允许目标由运行时凭证确定，不能信任命令行自报身份；CLI 默认输出适合人的摘要，并用 `--json` 或 stdin/stdout 提供稳定机器接口。

  - 测试：覆盖 CLI 帮助和稳定退出码、JSON 输入输出、代码观测分页与边界、控制面身份拒绝、CLI/API/工具结果一致性、不存在补丁子命令、Bash 直接修改候选后提交差异正确，以及大 diff 默认只输出摘要。

- [ ] 10. 实现异步 `execute_plan_node` 与持久化 wait 信号
  - 原因：当前 `backend/src/bridle/agent/runtime/gateway.py` 的 parent submit 路径使用内存 Future 等待 provider 完整返回，进程重启后无法恢复，也会把长时间节点工作流错误地绑定在一次工具调用上；`PersistentMailbox.wake_version` 只表示邮箱活动，不能作为某个工作流是否结束的事实源。
  - 修改：删除 `select_node` 的模型 schema、描述符和 handler，新增 `execute_plan_node(node_id)`。在项目本地计划事实源中增加节点工作流执行记录、wait 信号记录和 completion outbox；首次调用原子校验节点可运行性、创建执行与 `waiting` 信号、启动后台工作流并立即返回 `wait_id`、`execution_id`、`node_id`、`state` 和 `revision`。同一节点已有活动执行时必须复用原执行与 wait ID，不得重复创建候选或并行发布。
  - 修改：后台节点工作流负责执行前计划/代码地图与边界检查、测试编写、测试合同审核、适用场景的有效红、候选代码开发、最终测试、只读代码 Review、基线冲突检查、原子发布、`CodeChanged` 投递和代码地图追平。最终测试失败和 Review 拒绝只返回对应内部阶段，wait 继续保持 `waiting`；只有工作流进入 `completed`、`failed`、`blocked` 或 `cancelled` 终态时，才把 wait 设为 `ended` 并写入独立 `outcome`。
  - 修改：工作流终态事务必须同时写入 wait 终态、最小结果摘要和 completion outbox；outbox 随后幂等投递 `node-workflow-result` Mail 给主 Agent。扩展 `read_project_map` 的 `execution` 查询模式，允许按 `wait_id` 读取 wait 状态、当前 phase、revision 和终态 outcome；查询是纯读操作。进程内 Future、Task、`asyncio.Event` 和 mailbox wake signal 只可作为性能优化，不能成为 wait 状态事实源。
  - 保留：普通会话消息仍可使用现有同步 parent reply 语义，本项只改造计划节点执行入口；主 Agent 在工作流内部重试、测试失败或 Review 拒绝时不接收中间结果，超过预算或需要外部权限时工作流以终态 `blocked` 结束后再发送一次完成消息；wait 与 Mail 只携带身份、状态、摘要、错误码和必要引用，不传完整 diff、源码或命令记录。
  - 完成标准：`execute_plan_node` 在后台工作流尚未完成时就返回 `state=waiting`；服务重启后可按 wait ID 恢复查询和继续执行；内部循环不会把 wait 提前结束或向主 Agent发送终态消息；所有终态都会先持久化 `ended` 再投递且仅投递一次 Mail；Mail 暂时满或投递进程崩溃时 outbox 可重试；主 Agent从不查询也能收到最终消息，频繁查询也不会改变执行结果。
  - 测试：使用可控后台屏障证明工具在 worker 释放前已返回；覆盖同节点重复调用复用 wait、不同阶段查询、最终测试失败后继续等待、Review 拒绝后继续等待、成功/失败/阻塞/取消均正确结束、终态写入与 Mail 顺序、Mail 满后的持久化重试、终态消息去重、进程重启恢复、完全不查询仍收到结果，以及查询无副作用；禁止用单个立即完成的 fake 掩盖异步和竞态问题。
  - 验证：`python -m pytest backend/tests/unit/agent/runtime/test_node_workflow_wait_signal.py backend/tests/integration/agent/test_async_node_execution.py -q`

- 集成测试必须证明任意模型命令只能在候选容器执行，探索命令不能伪造门禁结果，宿主机回退路径不可达。

- 集成测试必须证明 `execute_plan_node` 即时返回可持久化 wait 信号，后台工作流跨重启继续，内部失败循环不通知主 Agent，终态先把 wait 设为 `ended` 再通过 Mail 唯一投递；主 Agent 查询或不查询都不影响最终结果。
